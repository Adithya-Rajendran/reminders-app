# CalDAV Re-platform Plan — tasks+reminders on CalDAV, SQLite config DB, retire Postgres

_Generated by a 6-architect design workflow (June 2026). Flag-gated (TASK_STORE, CONFIG_STORE), phased P0–P6, cutover gated._

# PLAN — Migrate Reminders App to CalDAV-backed Tasks + Reminders with a SQLite Config DB (retire Postgres)

This is the synthesized, ordered, implementation-ready plan reconciling all six component designs. Cross-component conflicts are resolved inline under **DECISION** callouts. Source files referenced are absolute under `/home/ubuntu/claude/reminders-app/`.

## 1. Summary + End-State Architecture

**Goal.** Move the canonical store for tasks/projects/labels/reminders/recurrence from Postgres into the user's own **CalDAV** server (VTODO + VALARM + RRULE), and move the small remaining **config** (dashboard layouts, CalDAV account creds, sessions, reminder bookkeeping) from Postgres into a single **SQLite** file (better-sqlite3, WAL) on a 1Gi ceph-rbd RWO PVC. Then delete Postgres. The SPA wire shape is preserved so client churn is minimal.

**End-state runtime.**
```
Browser SPA ──HTTP──> Express (single pod, replicas:1, Recreate)
   /api/{projects,tasks,labels}  ─► taskstore.js ─► tasks_caldav.js ──CalDAV──► user's server (VTODO)
   /api/calendar/events          ─► caldav.js (VEVENT, unchanged)
   /api/caldav/*  /api/layouts   ─► config-repo (sqlite impl) ─► /data/config.db (SQLite WAL)
   SSE "reminder"                ─► valarm-poller.js (reads reminder_cache, fires at-most-once)
                                     ▲ poller syncs CalDAV (sync-token/ctag) into SQLite reminder_cache
   sessions                      ─► better-sqlite3-session-store (same /data/config.db handle)
Postgres: DELETED
```

**Two orthogonal feature flags, both compiled into one image** (this is the backbone of the reversible rollout):

| Flag | Values | Ship default | Selects |
|---|---|---|---|
| `TASK_STORE` | `postgres` \| `caldav` | `postgres` | tasks/projects/labels/reminders/recurrence backend + which poller runs |
| `CONFIG_STORE` | `postgres` \| `sqlite` | `postgres` | layouts + CalDAV creds + sessions backend |

End state: `TASK_STORE=caldav`, `CONFIG_STORE=sqlite`, Postgres removed. **Cutover order is tasks-first, then config** (Postgres stays the intact rollback anchor through the higher-risk task flip; flipping config first would strand the `TASK_STORE=postgres` rollback path which needs the pg pool).

## 2. CalDAV Task / Project / Label Model

New module **`app/server/tasks_caldav.js`** (siblings of `tasks.js`), selected via **`app/server/taskstore.js`**. Reuses `caldav.js` helpers: `clientFor`, `getAccount`, `authHeader`, `safeFetch`, `assertEgressAllowed`, `enc/dec`, `VTODO_FILTER`, `discover`.

### 2.1 ID encoding — shared `app/server/taskid.js`
```js
const SEP = '\u001f'
const b64url   = (s) => Buffer.from(s,'utf8').toString('base64url')
const unb64url = (s) => Buffer.from(String(s),'base64url').toString('utf8')
export const encodeTaskId = (listId, objectUrl) => b64url(`${listId}${SEP}${objectUrl}`)
export function decodeTaskId(id){ /* b64 decode, split on SEP, Number(listId) must be integer, else 400 */ }
export const encodeLabelId = (name) => 'cat_' + b64url(name)
export const decodeLabelId = (id)   => unb64url(String(id).replace(/^cat_/,''))
```
- **`project_id` = integer `caldav_lists.id`** (NOT encoded). Positive, AUTOINCREMENT, never reused → `Dashboard.jsx:71/82` safe, unchanged.
- **`task.id` = `encodeTaskId(listId, objectUrl)`** (opaque string). `serializeVtodo` must **not** `Number()` it. `listId` gives O(1) ownership-checked resolution of account creds + list URL; `objectUrl` gives the exact href for GET/PUT/DELETE.
- **`label_id` = `cat_` + b64url(categoryName)** (CATEGORIES are free text, no server ids).
- The **reminder poller imports the same `encodeTaskId`** so RemindersWidget complete/snooze (`updateTask(t.id,…)`) round-trips correctly (D2).

### 2.2 `caldav_lists` schema (merged, D3/D4)
```sql
CREATE TABLE caldav_lists (
  id             INTEGER PRIMARY KEY AUTOINCREMENT,   -- = project_id (stable, never reused)
  account_id     TEXT NOT NULL REFERENCES caldav_accounts(id) ON DELETE CASCADE,
  url            TEXT NOT NULL,
  display_name   TEXT,
  color          TEXT,                                 -- → project.hex_color (Apple/NC calendar-color)
  supports_vtodo INTEGER NOT NULL DEFAULT 0,
  enabled        INTEGER NOT NULL DEFAULT 1,
  UNIQUE (account_id, url)                             -- ON CONFLICT keeps id stable on re-discovery
);
```
`discover()` extended to persist `color` (tsdav `calendarColor` / `{http://apple.com/ns/ical/}calendar-color`) and `supports_vtodo` (components include `VTODO`). Default list lives in `user_settings` (§5 schema).

### 2.3 projects = VTODO lists
`listProjects`: user's accounts → `caldav_lists WHERE supports_vtodo=1 AND enabled=1`, ordered `(url = default_list_url) DESC, display_name, id`. Emit:
```js
{ id: row.id, title: row.display_name||row.url, hex_color: row.color||'',
  is_inbox: row.url === defaultListUrl, description:'', parent_project_id:0 }
```
**`ensureDefaultList`** (must yield ≥1 project so `Dashboard.jsx:101 pr[0]` works): (1) if `user_settings.default_list_url` resolves to an enabled VTODO list, use it; (2) else set it to the lowest-`id` enabled VTODO list; (3) if zero VTODO lists, optionally `MKCALENDAR` a "Tasks" collection (`client.makeCalendar` with `supported-calendar-component-set:['VTODO']`) then re-`discover()`; if impossible, `listProjects` returns `[]` → SPA empty/onboarding state. No per-project create/rename/delete API (none exists today).

### 2.4 VTODO ↔ wire serializer (`serializeVtodo`)
Operate on the **master** VTODO (`pickMaster`: the one with no `RECURRENCE-ID`). `ZERO='0001-01-01T00:00:00Z'`, `outTs` reused from `tasks.js`.

| wire | VTODO | read | write (only if key ∈ patch body) |
|---|---|---|---|
| `id` | objectUrl+listId | `encodeTaskId(listId,o.url)` | n/a |
| `project_id` | containing list | `listId` | move = COPY+DELETE (§2.6) |
| `title` | `SUMMARY` | value or `'(untitled)'` | `summary` |
| `description` | `DESCRIPTION` | value or `''` | set / remove if `''` |
| `done` | `STATUS` | `===COMPLETED` | §2.5 |
| `done_at` | `COMPLETED`/`LAST-MODIFIED` | as table | set/remove with status |
| `due_date` | `DUE`??`DTSTART` | `toJSDate().toISOString()` else `ZERO` | §2.5 (preserve DATE typing) |
| `priority` | `PRIORITY` | `icalToOur` (§2.5) | `OUR_TO_ICAL` or remove if 0 |
| `repeat_after`/`repeat_mode` | `RRULE`/X-props | `repeatFieldsFromVtodo(vt)` (§3) | `applyRepeatFields` (§3) |
| `reminders[]` | `VALARM`s | `readReminders(vt)` (§4, D7) | `applyReminders` (§4) |
| `labels[]` | `CATEGORIES` | `readCategories` (§2.7) | `attachLabel`/`labels:[]` body |
| `created`/`updated` | `CREATED`/`LAST-MODIFIED`/`DTSTAMP` | `outTs(...)` | bump on every PUT |

### 2.5 Field details
- **done write** = `caldav.js` `toggleHandler` pattern: set `STATUS:COMPLETED`+`percent-complete:100`+`COMPLETED:now` / else `NEEDS-ACTION`+`0`+remove `completed`. Recurring completion (`done:false→true` while recurring) is delegated to recurrence `advanceRecurringVtodo` inside the same GET-modify-PUT.
- **due_date**: read `DUE??DTSTART`. On partial update **don't rewrite DUE unless `due_date` is in the body** (preserve imported `VALUE=DATE` all-day typing). On set, write timed UTC `DUE` (`ICAL.Time.fromJSDate(new Date(due_date), true)`, `resetType('date-time')`); `ZERO/''/null` → `removeAllProperties('due')`. Document westward day-shift on all-day reads.
- **priority map** (round-trip stable, monotonic):
```js
const OUR_TO_ICAL = {0:0,1:9,2:7,3:5,4:3,5:1}
function icalToOur(p){ const n=Number(p)||0
  if(n<=0)return 0; if(n<=2)return 5; if(n<=4)return 4; if(n<=6)return 3; if(n<=8)return 2; return 1 }
```
Write: `our===0`→remove `PRIORITY`; else `OUR_TO_ICAL[our]`.

### 2.6 CRUD (all egress via `safeFetch`, `redirect:'error'`)
- **createTask** `PUT /api/projects/:id/tasks` (`:id`=listId): resolve+own-check list; build VCALENDAR/VTODO (`uid=randomUUID()`, prodid, version, summary, status, optional description/priority/due, `applyReminders`, `applyRepeatFields`); `PUT objectUrl = listUrl/<uid>.ics` with **`If-None-Match:*`**; return `serializeVtodo`. Unknown/missing `project_id` → write to user's default list (per client-impact undo path), not 404.
- **patchTask** `POST /api/tasks/:id`: GET-modify-PUT in place. Only keys present in body are touched; everything else (foreign VALARMs, RRULE, X-*, ATTACH) round-trips untouched (ical.js preserves unknowns). `If-Match: etag`; on **412 re-GET+reapply once**, else `409`. Returns freshly serialized task (client inspects `r.done`/`r.due_date`). On `done:true && !cur.done && isRecurring(vt)` call `registerTimezones(vcal)` then `advanceRecurringVtodo(vt, now)` (§3). After success → `refreshCacheFromIcs` (D8).
- **move** (`project_id` change): create-at-target (`If-None-Match:*`) + delete-source → new `task.id`; client reloads. Calendar drag sends only `due_date` (no move).
- **deleteTask** `DELETE /api/tasks/:id`: GET etag → `DELETE` with `If-Match`; treat 204 and 404 as success (idempotent). After success → cache invalidate.

### 2.7 labels = CATEGORIES
No labels table. `readCategories(vt)`: union of all `CATEGORIES` multi-values → `[{id:encodeLabelId(title), title, hex_color:''}]` (dedup). `listLabels` = union across user's VTODOs (reuse §2.8 fan-out/cache). `createLabel` = echo encoded id (no server object). `attachLabel` / `labels:[{title}]` in patch body = GET-modify-PUT, add decoded category if absent, `If-Match`+412 retry. (Client switches to sending `labels:[{title}]` in the task body — see §6.)

### 2.8 Listing efficiency
- `listProjectTasks`: one list `fetchCalendarObjects({calendar:{url}, filters:VTODO_FILTER})` → serialize → sort `done ASC, due ASC, title`; slice `per_page`.
- `listTasks`: fan-out across all enabled VTODO lists with `Promise.allSettled` (skip failing list, like `fetchTasksHandler:250`), concat, sort by `SORT_COLS` (default `due_date ASC NULLS LAST` → push ZERO last), slice.
- **Per-user in-memory cache** `Map<sub,{at,lists:Map<url,{ctag,objects}>}>` ~10–15s TTL to coalesce the widget stampede (Upcoming + Calendar + every tasklist + SSE-triggered reloads). Invalidate the touched list after any user write. Optional ctag/sync-token skip of the REPORT. This cache is **independent** of the poller's persisted `reminder_cache` (§4); they may share `caldav_sync_state` later.

### 2.9 Backend selection — `app/server/taskstore.js`
```js
import * as pg from './tasks.js'
import * as cd from './tasks_caldav.js'
export const store = process.env.TASK_STORE === 'caldav' ? cd : pg
```
`index.js` imports `{ store as tasks }` — **routes (`:113–121`) and wire shape unchanged**. `tasks_caldav.js` exports the same 9 handlers: `listProjects, listProjectTasks, listTasks, createTask, patchTask, deleteTask, listLabels, createLabel, attachLabel`.

## 3. RRULE Recurrence Algorithm

New file **`app/server/recurrence_caldav.js`** (D9), pure ICS over an `ICAL.Component(vtodo)` — no DB/HTTP. ical.js 2.2.1. **Model decision: single evolving VTODO (date-shift on completion), NOT `RECURRENCE-ID` overrides** — matches Tasks.org/OpenTasks/DAVx5/Apple, round-trips with phones, and preserves the existing client contract (recurring completion returns `done:false` with advanced `due_date` → `useTasks.js` shows "Rescheduled ↻").

### 3.1 Primitives
`Recur#getNextOccurrence(start, after)` (returns first occurrence strictly after `after`, copies `after.zone`, **null when COUNT/UNTIL exhausted**); `Recur#isByCount/.freq/.interval/.count/.until`; `Time.fromJSDate/clone/compare/subtractDate→Duration/addDuration/isDate/zone`; `TimezoneService.register`.

### 3.2 `advanceRecurringVtodo(vt, now)` → `{advanced, done}` (mutates vt)
```
if hasCustomFromCompletion(vt): return advanceFromCompletion(vt, now)   // mode 2, §3.5
rrProp = vt.getFirstProperty('rrule'); if !rrProp: return {advanced:false,done:false}
anchorProp = vt.getFirstProperty('dtstart') || vt.getFirstProperty('due')
if !anchorProp: return {advanced:false,done:false}                       // no anchor → stay open
anchor = anchorProp.getFirstValue()
next = rrule.getNextOccurrence(anchor, anchor)                           // exactly one step
if !next:  // COUNT/UNTIL exhausted
   set STATUS:COMPLETED, percent-complete:100, COMPLETED:now; bumpRevision; return {advanced:false,done:true}
delta = next.subtractDate(anchor)
setAnchorTo(anchorProp, next)                                            // DST-correct wall time, preserve isDate
if dtstart && due: shiftBy(trailingProp, delta)
shiftAbsoluteAlarms(vt, delta)                                           // §4 / D7
if rrule.isByCount(): rrule.count -= 1; rrProp.setValue(rrule)          // UNTIL needs no bookkeeping
remove COMPLETED; STATUS:NEEDS-ACTION; percent-complete:0; bumpRevision
return {advanced:true, done:false}
```
`getNextOccurrence(anchor,anchor)` re-iterates from the *current* (moved) anchor, so COUNT decrement + UNTIL both terminate correctly. **Default = advance exactly one occurrence** (keeps overdue catch-up visible). `RECUR_SKIP_MISSED=1` opt-in loops until `next>now`.

### 3.3 Mode mapping
| legacy mode | store | advance reference |
|---|---|---|
| 0 from due | `FREQ=DAILY/WEEKLY/HOURLY/…;INTERVAL=n` | one step after current DUE/DTSTART |
| 1 monthly | `FREQ=MONTHLY;INTERVAL=n` | one step after anchor |
| 2 from completion | **no RRULE** — `X-REMINDERS-REPEAT-MODE/AFTER` | `now()+interval` |

### 3.4 Field mapping
- **Write `applyRepeatFields(vt, repeatAfterSec, repeatMode, anchorOpt)`** (= task-model's `applyRecurrence`, D11): clears RRULE+X-props; mode2→X-props; mode1→`FREQ=MONTHLY;INTERVAL=1`; else `secondsToFreq` picks coarsest exact unit (WEEKLY/DAILY/HOURLY/MINUTELY/SECONDLY). `repeat_after===0 && mode===0` clears. Always ensure an anchor (add `DTSTART`=due or now) when writing a new RRULE.
- **Read `repeatFieldsFromVtodo(vt)`**: X-props→mode2; RRULE freq×interval→seconds; MONTHLY→`{repeat_after:n*2629800, repeat_mode:1}`. **Badge gotcha (load-bearing):** `TaskRow.jsx` shows ↻ only when `repeat_after>0`, so MONTHLY must emit nonzero `repeat_after`. RRULE is source of truth; these fields are display-only and intentionally lossy (`BYDAY` etc. render as plain "weekly").

### 3.5 Mode 2 (`advanceFromCompletion`)
`newDue = now + X-REMINDERS-REPEAT-AFTER`; `delta = oldDue?newDue−oldDue:interval`; set DUE, `shiftBy(dtstart, delta)`, `shiftAbsoluteAlarms(vt, delta)`, reset to NEEDS-ACTION. Documented degradation: no RRULE → phone won't auto-advance (inherent; we advance server-side on completion).

### 3.6 VALARM shift (`shiftAbsoluteAlarms`, D7)
Relative `TRIGGER` (DURATION, RELATED=START/END) auto-shift with the moved anchor → **don't touch**. Absolute `TRIGGER;VALUE=DATE-TIME` → `shiftBy(delta)`. Wire reminders for the next occurrence come from the reminders module's `readReminders(vt)` after advance (not a separate function).

### 3.7 Tricky cases
COUNT decrement vs UNTIL fixed; **monthly end-of-month skips** (Jan31→Mar31, RFC; divergence from old overflow — document; `BYMONTHDAY=-1` for true last-day); all-day `VALUE=DATE` preserved; **register every `VTIMEZONE` via `TimezoneService.register` before iterating** (else DST drift) — caller passes the parsed VCALENDAR (`registerTimezones(vcal)` in patchTask); off-grid DUE snaps to next grid point; RRULE w/o anchor stays open; EXDATE honored, **RDATE not expanded** (out of scope v1, note `RecurExpansion` if needed); force NEEDS-ACTION/percent 0/strip COMPLETED on advance; 412 → re-GET + re-run.

### 3.8 Module API (`recurrence_caldav.js`)
`isRecurring(vt)`, `advanceRecurringVtodo(vt, now)`, `applyRepeatFields(...)`, `repeatFieldsFromVtodo(vt)`, `registerTimezones(vcal)`; internal `secondsToFreq, shiftAbsoluteAlarms, setAnchorTo, shiftBy, bumpRevision, hasCustomFromCompletion, advanceFromCompletion`. Old `recurrence.js` retired at P6.

## 4. VALARM Reminder Model + Poller + per-user SSE

New module **`app/server/valarm-poller.js`** (+ a reminders write helper, e.g. `valarm.js`); replaces `scheduler.js`. State moves to the SQLite config DB (shared `db` handle).

### 4.1 SSE envelope (byte-compatible with `scheduler.js`, D12)
Each `event: reminder` `data:` line:
```json
{"receivedAt":<ms>,"event":{"event_name":"task.reminder","data":{"task":{
  "id":"<encodeTaskId>","title":"...","due_date":"RFC3339|0001-...Z","priority":0,"done":false,
  "reminders":[{"reminder":"ISO"}]}}}}
```
`event_name ∈ {task.reminder, task.overdue}`. Reuse `reminderEvent()`/`outTs`/`ZERO` verbatim from `scheduler.js`. `task.id` = `encodeTaskId(list_id, object_url)` (D2) so widget complete/snooze round-trip.

### 4.2 Write path (D8)
- **`applyReminders(vt, remindersISO, {relateToDue})`** — pure in-VTODO mutator: removes only **our** alarms (tagged `X-REMINDERS-APP:1` + RFC 9074 `UID`), preserves foreign alarms, appends `VALARM`(s) (`ACTION:DISPLAY`, `DESCRIPTION:<summary>`, `TRIGGER`). Absolute → `ICAL.Time.fromJSDate(d,true)` (UTC `VALUE=DATE-TIME …Z`). `relateToDue` → relative `TRIGGER;RELATED=END:<duration>` (device auto-shifts + follows RRULE). Used by create/patch within their single GET-modify-PUT.
- **`setReminders(acc, objectUrl, remindersISO, opts)`** — thin GET→applyReminders→PUT wrapper with `If-Match` + 412×3 retry; on success calls `refreshCacheFromIcs`. Utility for snooze/poller.
- **`readReminders(vt)`** (D7) — VALARM → `[{reminder:ISO}]` (absolute → instant; relative → anchor+duration; RELATED=END uses DUE). Used by `serializeVtodo`.
- **Snooze** = `RemindersWidget.snooze` → `updateTask(id,{reminders:[{reminder:now+1h}]})` → patchTask → `applyReminders`. Old instant removed from `reminder_cache` by `refreshCacheFromIcs` → never re-fires. (Optional RFC 9074 `ACKNOWLEDGED`+`RELTYPE=SNOOZE` upgrade documented.)
- **`refreshCacheFromIcs(acc, objectUrl, ics, etag)`** — re-derive `reminder_cache` rows for that object immediately after our writes so the poller doesn't wait for sync.

### 4.3 `parseAlarms(ics, ctx, now, windowEnd)`
Per VTODO: absolute triggers (fire once); relative triggers (RELATED=END→DUE, START→DTSTART) — if RRULE, expand occurrences via `ICAL.RecurExpansion` (bounded by `LOOKAHEAD`, guard 1000); `REPEAT`/`DURATION` alarm-internal repeats fanned out; synthetic `alarm_id='__due__'` row carrying `due_at` so objects with DUE but no alarm still get overdue coverage. `alarm_id` = VALARM UID or SHA-1(objectUrl|i|trigger).

### 4.4 SQLite schema (in the config DB; D6, + `list_id` per D2)
```sql
CREATE TABLE IF NOT EXISTS caldav_sync_state (
  account_id TEXT, list_url TEXT, sync_token TEXT, ctag TEXT, last_sync_at INTEGER,
  PRIMARY KEY (account_id, list_url));

CREATE TABLE IF NOT EXISTS reminder_cache (
  user_id TEXT, account_id TEXT, list_id INTEGER, list_url TEXT, object_url TEXT,
  uid TEXT, alarm_id TEXT, remind_at INTEGER NOT NULL, due_at INTEGER, etag TEXT,
  summary TEXT, priority INTEGER DEFAULT 0, done INTEGER DEFAULT 0,
  PRIMARY KEY (object_url, alarm_id, remind_at));
CREATE INDEX IF NOT EXISTS reminder_cache_fire ON reminder_cache(remind_at) WHERE done=0;

CREATE TABLE IF NOT EXISTS fired_reminders (
  user_id TEXT, object_url TEXT, alarm_id TEXT, remind_at INTEGER, fired_at INTEGER,
  PRIMARY KEY (object_url, alarm_id, remind_at));   -- alarm_id='__overdue__' for DUE sweep
CREATE INDEX IF NOT EXISTS fired_reminders_gc ON fired_reminders(fired_at);
```
Composite PK = **at-most-once per fire instant**. better-sqlite3 is synchronous single-process → `SELECT…then INSERT` inside one `db.transaction` is the atomic claim (no `SKIP LOCKED`). `list_id` lets `firePass` build `encodeTaskId(list_id, object_url)`.

### 4.5 Poll loop (two cadences, non-overlapping `ticking` guard)
- **Sync pass** per list every `SYNC_INTERVAL` (default 300s): RFC 6578 `sync-collection` REPORT (tsdav `smartCollectionSync`) using stored `sync_token`; fallback to `CS:getctag` for old servers (full crawl on ctag change). Only changed hrefs → `calendar-multiget` → re-derive `reminder_cache` rows; removed hrefs → delete cache+fired rows. First run = initial full sync. Bounded fan-out `p-limit(4)` + per-list jitter.
- **Fire pass** every `FIRE_INTERVAL` (default 30s, `REMINDER_POLL_MS`): **no network**. Select `reminder_cache WHERE done=0 AND remind_at<=now AND NOT EXISTS(fired)` LIMIT 500; in a txn emit SSE (only if `remind_at >= now − CATCHUP_MS`, else silent-claim) and insert `fired_reminders`. Then the **overdue sweep** (`due_at<=now`, `alarm_id='__overdue__'`) for parity with `scheduler.js`. `reminderEvent` lifted unchanged.
- **Top-level multi-user loop**: walks all `caldav_accounts` (global, not per-request), decrypts creds with `dec()`/`authHeader()`, targets `acc.user_id` via `sendToUser`, egress via `safeFetch`/`assertEgressAllowed`. `startValarmPoller()` runs `firePass(now)` immediately at boot (catch-up from persisted cache) then `setInterval(tick, FIRE_INTERVAL).unref()`.

Defaults: `FIRE_INTERVAL=30000`, `SYNC_INTERVAL=300000`, `LOOKAHEAD=7d`, `CATCHUP_MS=24h`, `CONCURRENCY=4`. Steady-state cost ≈ `U·L/SYNC_INTERVAL` (homelab: ~0.1 req/s).

### 4.6 GC / edges
`done=1` rows excluded; `CATCHUP_MS` bounds outage flood (silent-claim older); `gcList(accountId,listUrl)`/`gcAccount(accountId)` called by `setListsHandler`/`deleteAccountHandler`; `DELETE fired_reminders WHERE fired_at<now−30d` each tick; foreign alarms preserved + de-duped by alarm_id; absolute triggers always UTC + DISPLAY needs DESCRIPTION.

### 4.7 Flag gating (D12) — `index.js`
```js
if ((process.env.TASK_STORE||'postgres')==='caldav') startValarmPoller()
else startScheduler()
```
Both modules stay in the tree for reversible cutover. No SSE feed gap.

## 5. SQLite Config DB

### 5.1 Library — **better-sqlite3** (`^11.8.1`) + **better-sqlite3-session-store** (`^0.1.0`)
Synchronous, battle-tested, first-class session store; ships glibc x64/arm64 prebuilds (base image `node:22-bookworm-slim`). Reject experimental `node:sqlite` (no maintained session store). Add `python3 make g++` to the **build** stage of `app/Dockerfile` only (defensive; runtime stays slim, copies `node_modules`). `package.json`: add the two deps; remove `pg` + `connect-pg-simple` **at P6**.

### 5.2 Config-repo abstraction + `CONFIG_STORE` flag (D5)
New **`app/server/config-repo.js`** with a tiny surface, two impls (pg = existing SQL; sqlite = better-sqlite3 native `?` params), selected at boot:
```js
const cfg = (process.env.CONFIG_STORE||'postgres')==='sqlite'
  ? makeSqliteConfig(process.env.CONFIG_DB_PATH||'/data/config.db')   // opens db, initConfigSchema()
  : makePgConfig(pool)
```
Repos: `layoutsRepo{get,save}`, `accountsRepo{listByUser,get,insert,delete}`, `listsRepo{listFor,upsert,setEnabled,pruneNotIn}`, `settingsRepo{get,setDefaultList}`, plus the flag-selected session store. `caldav.js` is refactored to call repo methods instead of raw `pool.query` (this is where the dialect differences—`$n`→`?`, `now()`→`strftime`, `<>ALL($2::text[])`→`pruneNotIn` `NOT IN (placeholders)`, `enabled` 0/1 vs boolean, avoid `RETURNING`—are isolated per impl). `getLayout`/`saveLayout` become `layoutsRepo` (JSON.parse/stringify the `layout_json` TEXT). Shared raw `db` handle exported from `db.js`.

### 5.3 Unified SQLite schema (idempotent `initConfigSchema`, `db.exec`)
- `user_dashboards(user_id,dashboard_id,layout_json TEXT,layout_version INT,updated_at TEXT, PK(user_id,dashboard_id))`
- `caldav_accounts(id PK,user_id,name,type,server_url,username,password_enc,created_at)` + idx on user_id
- `caldav_lists(...)` — **full schema from §2.2** (id AUTOINCREMENT, color, supports_vtodo, FK cascade)
- `user_settings(user_id PK, default_account_id, default_list_url)` — onboarding/default (D4)
- `caldav_sync_state`, `reminder_cache`, `fired_reminders` — **from §4.4** (D6)
- `_config_migrations(id PK, applied_at)`
- sessions table auto-created by `better-sqlite3-session-store`.

Type mapping: `jsonb`→TEXT, `boolean`→0/1, `timestamptz`→ISO TEXT.

### 5.4 Pragmas (at open)
`journal_mode=WAL`, `busy_timeout=5000`, `synchronous=NORMAL`, `foreign_keys=ON`. WAL is safe here because the file is on **ceph-rbd block** (real ext4/xfs, honors `fcntl` POSIX locks + mmap for `-shm`); single replica + `Recreate` + RWO means one process ever opens it. **Never CephFS/NFS.**

### 5.5 Session store (`index.js`)

Cookie config unchanged. **Sessions not migrated** — one forced re-login on the `CONFIG_STORE` flip. Drop `caldav.initCaldavDb()` (folded into `initConfigSchema`).

### 5.6 Kubernetes (`k8s/`)

## 7. Migration + Cutover + Postgres Retirement

### 7.1 What moves (config only)
Copy → SQLite: `user_dashboards` (jsonb→TEXT), `caldav_accounts` (`password_enc` **ciphertext verbatim**), `caldav_lists`. **Not migrated:** sessions (re-auth once), pg task tables (no canonical data — retired with PG). `fired_reminders`/`reminder_cache` start empty (poller rebuilds via initial sync).

### 7.2 `app/server/migrate-config.js` (idempotent, dry-run)
Reads PG (`SELECT *` from the 3 tables), upserts into SQLite via `ON CONFLICT … DO UPDATE excluded.*` keyed on real PKs (re-runnable, converges, never deletes from PG), wraps in one `db.transaction`, stamps `_config_migrations('pg->sqlite-config-v1')`. `--dry-run` prints counts. Run: `kubectl -n reminders-app exec deploy/reminders-app -- node server/migrate-config.js --dry-run`, reconcile vs `psql` counts, then run for real, then **re-run immediately before the `CONFIG_STORE` flip** to capture late edits (config writes are rare → no dual-write / no hard freeze).

### 7.3 Migration caveats / preconditions
- **R1 — `CALDAV_ENC_KEY` continuity:** ciphertext is AES-256-GCM keyed by `sha256(CALDAV_ENC_KEY||SESSION_SECRET||'dev-insecure')` (`caldav.js:55`). **Pin an explicit `CALDAV_ENC_KEY` (P3) before any migration** and verify a decrypt round-trip in the gate — a rotated `SESSION_SECRET` would brick every connected account.
- **D9 — stale-layout projectId collision (decision callout):** old pg layouts hold integer projectIds; new `caldav_lists.id` are independent small ints → a tasklist widget could silently point at the *wrong* list (valid data, no 404, no heal). **Mitigation in `migrate-config.js`:** when copying `user_dashboards`, null out / invalidate each `tasklist` widget's `projectId` so it heals to `pr[0]` (the default list) on first load. (Single-user homelab — deterministic and trivial.)
- `layout_json` jsonb → `JSON.stringify` to TEXT.

### 7.4 Cutover sequencing (tasks-first, both flips reversible)
1. **P4 tasks:** `pg_dump app` backup → flip `TASK_STORE=caldav` → `rollout restart` → onboarding gate confirms `ready` → soak 3–7 days. Postgres fully intact.
2. **P5 config:** `migrate-config.js --dry-run` → reconcile → re-run for real → flip `CONFIG_STORE=sqlite` → `rollout restart` (users re-auth once) → soak.

### 7.5 Verification gate (after each flip; test-harness: API via `kubectl exec`, UI via docker Playwright on `tasks.rke.alaras.net`, no port-forward, real session)
After P4: `GET /api/caldav/status ready:true`; `/api/projects` lists VTODO lists (default first); create-with-no-project lands in default list (verify VTODO in Nextcloud); toggle done edits in place (foreign VALARM preserved); `/api/tasks?sort_by=due_date` keeps wire shape (ZERO sentinel, `reminders:[{reminder}]`); recurring complete returns `done:false` + advanced due; SSE `reminder` arrives. After P5: saved layout resolves (proves `user_dashboards` copy), accounts list from SQLite, **decrypt round-trip** (`/api/caldav/tasks` returns tasks ⇒ key survived), list enable/disable. UI smoke: dashboard + task widget + calendar render, Settings shows account.

### 7.6 Rollback per phase
- **P0–P3:** revert image/manifest; PG untouched.
- **P4:** `TASK_STORE=postgres` + restart (pg store returns, empty — CalDAV tasks stay in Nextcloud; break-glass for app bug, not data recovery).
- **P5:** `CONFIG_STORE=postgres` + restart; PG holds pre-migration config; post-migration config edits lost unless `migrate-config.js` re-run (idempotent) — keep the window short, prefer fix-forward; users re-auth.
- **P6:** no clean rollback — only after clean soak; final `pg_dump` + SQLite file are last resort.

### 7.7 Postgres retirement (P6 — hard-gated on `TASK_STORE=caldav AND CONFIG_STORE=sqlite AND clean soak`)
Final `pg_dump app` copied out. **k8s:** delete `k8s/10-postgres.yaml`, PVC `data-pg-0`, secret `reminders-pg`; remove `DATABASE_URL` from `reminders-app-env`; delete/keep `50-networkpolicy.yaml`. **Code:** delete `app/server/{tasks.js,recurrence.js,scheduler.js}`; in `index.js` remove `startScheduler()`, pg task imports, and the now-CalDAV-served route block; remove `connect-pg-simple`; in `db.js` drop the pg pool + pg task DDL (becomes the SQLite repo). **`package.json`:** drop `pg`+`connect-pg-simple`. **Docs/memory:** rewrite `README.md` (architecture, env table — drop `DATABASE_URL`, add `CONFIG_DB_PATH`, note SQLite-on-RWO-block + `strategy:Recreate`); update `reminders-app-project.md` (tasks/reminders now VTODO/VALARM, config SQLite, PG retired) and `reminders-app-test-harness.md` (config at `/data/config.db`, inspect via `kubectl exec … sqlite3`).

### 8.1 Riskiest parts + de-risk
- **R1 — Enc-key continuity (highest).** Undecryptable accounts if `CALDAV_ENC_KEY` isn't byte-identical. *De-risk:* pin explicit key in P3; assert decrypt round-trip in every gate; migrate ciphertext verbatim.
- **R2 — SQLite storage/locking/rollout.** *De-risk:* ceph-rbd RWO block only (never CephFS/NFS); `replicas:1`; **`strategy:Recreate`** (avoid multi-attach deadlock); `fsGroup:1000` (writable mount); WAL + `busy_timeout`.
- **R3 — Onboarding gate.** Flipping `TASK_STORE` for a user with no account/default list breaks the task UI. *De-risk:* per-user readiness checklist before P4; `409 onboarding_required` + forced Settings gate; default-list auto-pick in `ensureDefaultList`.
- **R4 — Two-flag retirement ordering.** Deleting PG while either flag points at it = outage/data loss. *De-risk:* P6 hard-gated on both flags + clean soak.
- **R5 — Stale-layout projectId mispoint (D9, integer-id consequence).** *De-risk:* invalidate `tasklist` widget projectIds during `migrate-config.js` so they heal to the default list.
- **R6 — Recurrence/timezone correctness.** DST drift / monthly EOM divergence / COUNT-UNTIL termination. *De-risk:* `registerTimezones(vcal)` before iterating; unit tests on COUNT=1/3, UNTIL, all-day, monthly-31, DST-boundary; document EOM-skip behavior change.
- **R7 — At-most-once under recurrence/snooze.** *De-risk:* composite PK `(object_url,alarm_id,remind_at)` + `refreshCacheFromIcs` on every write + `CATCHUP_MS` flood bound; distinct `remind_at` per occurrence.
- **R8 — Session reset on P5 (low).** One forced re-login; document.
