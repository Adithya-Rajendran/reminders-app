# PLAN — Re-platform Reminders to Postgres-native tasks; retire Vikunja

## 0. Summary

Replace the Vikunja reverse-proxy (`vikunja.js`) with a Postgres-native task/project/label store and an in-app reminder scheduler, fixing the shared-service-account cross-user data leak and the "POST = full-object replace" footgun. The defining constraint is **continuity**: the live single-pod app at `tasks.rke.alaras.net` must never break, and the cutover must be reversible by flipping one env var. To get that, the new backend is built to be **byte-compatible with Vikunja's wire contract** so it can be served under the *same* `/api/vikunja/*` routes the client already calls — a `TASKS_BACKEND` flag selects proxy vs native. **The cutover therefore needs zero client changes.** A later, optional cleanup phase modernizes routes/verbs/SSE and removes the "vikunja" naming.

### Conflicts resolved across the component designs (deliberate decisions)

| # | Conflict | Resolution |
|---|---|---|
| C1 | owner column: `user_id` vs `owner_sub` | **`user_id text`** everywhere (matches existing `user_dashboards`/`caldav_*`; OIDC `sub`). |
| C2 | `GENERATED ALWAYS` identity vs preserving Vikunja IDs | **`GENERATED BY DEFAULT AS IDENTITY`** — allows explicit-ID migration inserts + `setval`, auto-gen for new rows. |
| C3 | schema says "greenfield, no migration" vs full migration | **Full ID-preserving migration.** `user_dashboards.layout_json` persists `tasklist` widgets as `{projectId:<vikunja id>}` (`Dashboard.jsx:82`); renumbering silently breaks every saved dashboard. Greenfield is rejected. |
| C4 | client: repoint+modern-verbs vs keep-grammar vs zero-change | **Two-stage.** Cutover serves native handlers on the *existing* `/api/vikunja/*` verb grammar → **zero client diff + invisible flag-flip rollback**. Optional Phase 6 modernizes. |
| C5 | SSE event name `tasks`/`reminder` + flat envelope vs keep `vikunja` nested | **Keep `vikunja` event + nested envelope at cutover** (byte-compatible, zero client diff). Rename/flatten only in optional Phase 6. |
| C6 | `repeat_mode`: brief's swapped description vs real Vikunja enum | **Real enum: 0=from-due, 1=monthly, 2=from-current.** The recurrence design is authoritative; migration copies integers verbatim. |
| C7 | recurrence trigger `repeat_after>0` vs `… or repeat_mode==1` | **Trigger on `repeat_after>0 OR repeat_mode==1`** (monthly may carry interval 0). |
| C8 | empty `due_date` serialize `null` vs zero sentinel | **Store NULL; emit zero sentinel `0001-01-01T00:00:00Z`.** Accept null/''/sentinel on write→NULL. Required for Vikunja byte-compat during dual-run. |
| C9 | `task_reminders` column `reminder` vs `remind_at`; `fired_at`/`user_id` presence | **Unified:** `id, task_id, user_id (denorm, NOT NULL), remind_at, fired_at, created_at`; JSON key stays `reminder` via serializer. |
| C10 | simple FKs vs composite `(child,user_id)→parent(id,user_id)` | **Composite FKs** — structural multi-tenancy fix (DB-enforced, not just app code). |
| C11 | scheduler robustness (deferred to design) | **Single-replica 30s poller + `fired_at` atomic claim + boot catch-up + at-most-once delivery.** Justified in §4. |
| C12 | Inbox at literal `id=1` vs `is_inbox` flag | **`is_inbox` flag + lazy `ensureInbox`.** Don't depend on id=1; the client `|| 1` is dead code (hardened in Phase 6). |

## 1. Consolidated SCHEMA (final DDL)

Add to `initDb()` in `app/server/db.js` (idempotent; co-located with the existing `user_dashboards`). All `CREATE … IF NOT EXISTS`. Register the int8 parser once at module load so bigint ids come back as JS numbers (ids stay < 2⁵³): `pg.types.setTypeParser(20, v => v===null ? null : Number(v))`.

```sql
CREATE OR REPLACE FUNCTION set_updated_at() RETURNS trigger AS $$
BEGIN NEW.updated_at = now(); RETURN NEW; END $$ LANGUAGE plpgsql;

-- projects -------------------------------------------------------------
CREATE TABLE IF NOT EXISTS projects (
  id          bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  user_id     text NOT NULL,
  title       text NOT NULL CHECK (length(title) BETWEEN 1 AND 250),
  hex_color   text NOT NULL DEFAULT '',
  is_inbox    boolean NOT NULL DEFAULT false,
  position    double precision NOT NULL DEFAULT 0,
  created_at  timestamptz NOT NULL DEFAULT now(),
  updated_at  timestamptz NOT NULL DEFAULT now(),
  UNIQUE (id, user_id)                              -- composite-FK target
);
CREATE INDEX IF NOT EXISTS projects_user_idx ON projects(user_id);
CREATE UNIQUE INDEX IF NOT EXISTS projects_one_inbox_per_user
  ON projects(user_id) WHERE is_inbox;             -- ensureInbox conflict target

-- tasks ----------------------------------------------------------------
CREATE TABLE IF NOT EXISTS tasks (
  id           bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  user_id      text NOT NULL,
  project_id   bigint NOT NULL,
  title        text NOT NULL CHECK (length(title) BETWEEN 1 AND 1000),
  description  text NOT NULL DEFAULT '',
  done         boolean NOT NULL DEFAULT false,
  done_at      timestamptz,
  due_date     timestamptz,                         -- NULL = no date (zero-sentinel on wire)
  priority     smallint NOT NULL DEFAULT 0 CHECK (priority BETWEEN 0 AND 5),
  repeat_after integer  NOT NULL DEFAULT 0 CHECK (repeat_after >= 0),  -- seconds
  repeat_mode  smallint NOT NULL DEFAULT 0 CHECK (repeat_mode IN (0,1,2)),
  position     double precision NOT NULL DEFAULT 0,
  overdue_notified_at timestamptz,                  -- scheduler overdue de-dupe
  created_at   timestamptz NOT NULL DEFAULT now(),
  updated_at   timestamptz NOT NULL DEFAULT now(),
  UNIQUE (id, user_id),
  FOREIGN KEY (project_id, user_id)
    REFERENCES projects(id, user_id) ON DELETE CASCADE,
  CHECK ((done=false AND done_at IS NULL) OR (done=true AND done_at IS NOT NULL))
);
CREATE INDEX IF NOT EXISTS tasks_user_project_idx ON tasks(user_id, project_id);
CREATE INDEX IF NOT EXISTS tasks_user_due_idx     ON tasks(user_id, due_date);
CREATE INDEX IF NOT EXISTS tasks_user_open_due_idx
  ON tasks(user_id, due_date) WHERE done=false AND due_date IS NOT NULL;

-- labels ---------------------------------------------------------------
CREATE TABLE IF NOT EXISTS labels (
  id         bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  user_id    text NOT NULL,
  title      text NOT NULL CHECK (length(title) BETWEEN 1 AND 250),
  hex_color  text NOT NULL DEFAULT '',
  created_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE (id, user_id)
);
CREATE INDEX IF NOT EXISTS labels_user_idx ON labels(user_id);
CREATE UNIQUE INDEX IF NOT EXISTS labels_user_title_ci ON labels(user_id, lower(title));

-- task_labels (join) ---------------------------------------------------
CREATE TABLE IF NOT EXISTS task_labels (
  task_id  bigint NOT NULL,
  label_id bigint NOT NULL,
  user_id  text   NOT NULL,
  PRIMARY KEY (task_id, label_id),
  FOREIGN KEY (task_id,  user_id) REFERENCES tasks(id, user_id)  ON DELETE CASCADE,
  FOREIGN KEY (label_id, user_id) REFERENCES labels(id, user_id) ON DELETE CASCADE
);
CREATE INDEX IF NOT EXISTS task_labels_label_idx ON task_labels(label_id);

-- task_reminders -------------------------------------------------------
CREATE TABLE IF NOT EXISTS task_reminders (
  id         bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  task_id    bigint NOT NULL,
  user_id    text   NOT NULL,                       -- denormalized for the scheduler
  remind_at  timestamptz NOT NULL,
  fired_at   timestamptz,                           -- NULL = pending; set on claim
  created_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE (task_id, remind_at),
  FOREIGN KEY (task_id, user_id) REFERENCES tasks(id, user_id) ON DELETE CASCADE
);
CREATE INDEX IF NOT EXISTS task_reminders_pending_idx
  ON task_reminders(remind_at) WHERE fired_at IS NULL;
CREATE INDEX IF NOT EXISTS task_reminders_task_idx ON task_reminders(task_id);
```

Triggers (no `IF NOT EXISTS` pre-PG14): wrap in `DROP TRIGGER IF EXISTS … ; CREATE TRIGGER … FOR EACH ROW EXECUTE FUNCTION set_updated_at();` for `projects` and `tasks`.

Notes: composite FKs make it structurally impossible to attach your task to another user's project/label (the core tenancy fix). `GENERATED BY DEFAULT` is mandatory (not `ALWAYS`) so the migration can insert preserved ids.

### Row → wire serializer (single source of truth)

```js
const ZERO = '0001-01-01T00:00:00Z'
const outTs = d => (d && new Date(d).getUTCFullYear() > 1) ? new Date(d).toISOString() : ZERO
function serializeTask(t, labels=[], reminders=[]) {
  return {
    id: Number(t.id), project_id: Number(t.project_id),
    title: t.title, description: t.description || '',
    done: t.done, done_at: outTs(t.done_at), due_date: outTs(t.due_date),
    priority: t.priority, repeat_after: t.repeat_after, repeat_mode: t.repeat_mode,
    reminders: reminders.map(r => ({ reminder: new Date(r.remind_at).toISOString() })),
    labels: labels.map(l => ({ id: Number(l.id), title: l.title, hex_color: l.hex_color })),
    created: outTs(t.created_at), updated: outTs(t.updated_at),
  }
}
```

Canonical read projection (used by every read endpoint **and** as the `RETURNING` re-projection after writes, so create/patch return a full task object):

```sql
SELECT t.id, t.title, t.description, t.done, t.done_at, t.due_date, t.priority,
       t.repeat_after, t.repeat_mode, t.project_id, t.created_at, t.updated_at,
       COALESCE(l.labels,'[]'::json)    AS labels,
       COALESCE(r.reminders,'[]'::json) AS reminders
FROM tasks t
LEFT JOIN LATERAL (
  SELECT json_agg(json_build_object('id',lb.id,'title',lb.title,'hex_color',lb.hex_color)
                  ORDER BY lower(lb.title)) AS labels
  FROM task_labels tl JOIN labels lb ON lb.id=tl.label_id WHERE tl.task_id=t.id) l ON true
LEFT JOIN LATERAL (
  SELECT json_agg(json_build_object('reminder', tr.remind_at) ORDER BY tr.remind_at) AS reminders
  FROM task_reminders tr WHERE tr.task_id=t.id) r ON true
WHERE t.user_id=$1 AND t.project_id=$2
ORDER BY t.done, t.position, t.id LIMIT 250;
```

Upcoming variant: `WHERE t.user_id=$1 AND NOT t.done AND t.due_date IS NOT NULL ORDER BY t.due_date ASC NULLS LAST LIMIT 250` (served by `tasks_user_open_due_idx`).

`ensureInbox(userId)`:

## 3. Recurrence algorithm

New `app/server/recurrence.js`. Pure function `nextOccurrence(old, now)`, evaluated on the **current stored row** when `done` transitions false→true and `(repeat_after>0 || repeat_mode===1)`. Enum: **0 = from due_date, 1 = monthly (calendar), 2 = from current time** (the real Vikunja meaning — the task brief's description is swapped; honoring the DB integers keeps migrated tasks correct).

```js
const isReal = d => d instanceof Date && !isNaN(d)
const addSeconds = (d,s) => new Date(d.getTime() + s*1000)
// Go AddDate(0,1,0)-compatible: same DoM next month, overflow normalized
// (Jan 31 -> Mar 3), computed in UTC to avoid DST drift.
const addOneMonth = d => new Date(Date.UTC(
  d.getUTCFullYear(), d.getUTCMonth()+1, d.getUTCDate(),
  d.getUTCHours(), d.getUTCMinutes(), d.getUTCSeconds(), d.getUTCMilliseconds()))

function nextOccurrence(old, now = new Date()) {
  const { repeat_after: interval, reminders } = old           // reminders: Date[]
  const oldDue = isReal(old.due_date) ? old.due_date : null
  const mode = [0,1,2].includes(old.repeat_mode) ? old.repeat_mode : 0

  if (mode === 1) {                                            // monthly
    if (!oldDue) return { advanced:false, due_date:null, reminders }
    return { advanced:true, due_date:addOneMonth(oldDue), reminders:reminders.map(addOneMonth) }
  }
  if (mode === 2) {                                            // from current — only mode that advances w/o a due date
    const newDue = addSeconds(now, interval)
    const shift = oldDue ? (newDue - oldDue) : interval*1000
    return { advanced:true, due_date:newDue, reminders:reminders.map(r=>new Date(r.getTime()+shift)) }
  }
  if (!oldDue) return { advanced:false, due_date:null, reminders }   // mode 0, no anchor
  return { advanced:true, due_date:addSeconds(oldDue,interval), reminders:reminders.map(r=>addSeconds(r,interval)) }
}
```

Integration in `patchTask` (transactional): on a recurring completion, **suppress the done flip** — write `done=false, done_at=NULL`; if `advanced` write the new `due_date` + replace `task_reminders` with the shifted set (`fired_at=NULL`); if not advanced (mode 0/1 with no due date), change nothing about due/reminders. Always return the full task with `done:false`. This exactly satisfies the client contract: `useTasks.onToggle` (line 53-59) and `RemindersWidget.complete` (line 49) branch on `r.done===false`, then compute `advanced = isRealDate(r.due_date) && (!isRealDate(task.due_date) || new(r.due)>new(task.due))` to show "Rescheduled ↻" vs the honest "Recurring — set a due date to complete". **Do not** do year-0001 arithmetic (Vikunja's bug) — null stays null unless a mode genuinely yields a real future date. Unit-test the 12-row matrix from the recurrence design (default/monthly/from-current × due/no-due, Go month overflow `2026-01-31→2026-03-03`, corrupt mode→0, done→done no-op).

## 4. Reminder scheduler — design + robustness recommendation

New `app/server/scheduler.js`. Replaces Vikunja cron→webhook→SSE.

**Recommendation (the deferred decision): a single-replica `setInterval` poller (30s, `REMINDER_POLL_MS`) with a `fired_at` atomic claim, automatic boot catch-up, and at-most-once delivery.** Justification:

- **Not bare fire-and-forget:** `fired_at` gives idempotent de-dupe and makes catch-up correct after a pod restart.
- **Not a durable job queue/outbox:** this is a single-pod homelab app with human-scale reminder volume (dozens, not millions); delivery receipts would be over-engineering.
- **Catch-up is free:** the steady-state predicate `remind_at <= now() AND fired_at IS NULL` already covers everything that came due while the pod was down — one immediate tick at boot fires it; a bounded drain loop clears a large backlog within the tick.
- **At-most-once is the right tradeoff for a notification feed:** the **task list is the durable source of truth** — a missed buzz still shows as an overdue chip (`dueChip` renders `overdue`). Occasionally missing a buzz beats duplicate buzzes on every crash. The optional `/api/reminders/recent` backfill (last 6h of fired reminders, queried on EventSource connect) closes the "fired while disconnected" gap if desired.

Claim + dispatch (atomic; `SKIP LOCKED` makes it safe under an overrunning tick or a future second replica):

```sql
WITH due AS (
  SELECT id FROM task_reminders
   WHERE fired_at IS NULL AND remind_at <= now()
   ORDER BY remind_at FOR UPDATE SKIP LOCKED LIMIT 500)
UPDATE task_reminders r SET fired_at = now()
  FROM due, tasks t WHERE r.id = due.id AND t.id = r.task_id
RETURNING r.task_id, r.user_id, r.remind_at, t.title, t.due_date, t.done, t.priority;
```

Skip `done` rows; for each, `sendToUser(user_id, 'vikunja', { receivedAt: Date.now(), event: { event_name:'task.reminder', data:{ task:{ id, title, due_date(ISO or ZERO), priority, reminders:[{reminder: remind_at}] } } } })`. Overdue sweep (separate statement, same tick) claims `done=false AND due_date<=now() AND (overdue_notified_at IS NULL OR overdue_notified_at < due_date)`, sets `overdue_notified_at=now()`, emits `event_name:'task.overdue'`. All comparisons use DB `now()` (no Node clock skew). Re-entrancy guard (`ticking`) + `timer.unref()`. Gate `startScheduler()` on `TASKS_BACKEND==='postgres'` so it's dormant pre-flip and on rollback.

**Envelope is byte-compatible with the old Vikunja webhook** (`event.data.task`, `event_name` containing "reminder"/"overdue", SSE event name `vikunja`) → `RemindersWidget` and `Dashboard.jsx:105` need **zero change** at cutover. Multi-replica caveat: the claim/de-dupe path is already replica-safe; only SSE dispatch is pinned to one pod — documented upgrade path is Postgres `LISTEN/NOTIFY` fan-out. Keep `replicas:1, strategy:Recreate`.

## 7. Data migration + cutover + Vikunja retirement (with rollback)

New script `app/server/migrate-vikunja.js` (ESM, run inside the pod; reuses `pool` + a Vikunja service-account `login()`). **Idempotent, ID-preserving, transactional**, run as `kubectl -n reminders-app exec deploy/reminders-app -- node server/migrate-vikunja.js --user <sub> [--dry-run] [--include-done]`.

**Whose `user_id`?** All Vikunja data is under one service account → every row gets the **primary human's OIDC `sub`**, passed explicitly via `--user` (never guessed). Discover the candidate from `SELECT user_id,count(*) FROM user_dashboards GROUP BY 1 ORDER BY 2 DESC` and cross-check `session` (`sess->'user'->>'sub'`).

**Steps inside one txn:** (a) projects — skip `id<=0` pseudo-projects (client already filters `p.id>0`); `INSERT … ON CONFLICT (id) DO UPDATE` preserving ids; mark the Vikunja default/Inbox project `is_inbox=true` (else `ensureInbox` synthesizes one). (b) labels — global→per-user, ids preserved. (c) tasks — page `per_page=100&page=N` via `x-pagination-total-pages`, **include done** (`filter=done = true || done = false`) for history/`done_at`; map zero-sentinel/'' → NULL for `due_date`/`done_at`; clamp priority; copy `repeat_after`/`repeat_mode` **verbatim**. (d) reminders — delete-then-insert per task; resolve purely-relative reminders to absolute against due/start; **set `fired_at=now()` where `remind_at <= now()`** so the scheduler's boot tick does **not** flood the user with historical buzzes — only future reminders stay pending. (e) task_labels from embedded `t.labels`, upserting any label not in `/labels`. (f) `setval` each identity sequence to `GREATEST(max(id),1)` so new rows don't collide with preserved ids. `--dry-run` wraps in `BEGIN…ROLLBACK` and prints counts. **Also emit a sample report of `repeat_mode<>0` tasks** to verify the 1=monthly/2=from-current mapping against live Vikunja before flip (R3).

**Cutover sequencing (app never broken; the flip is one env var):**
1. Ship dual-capable image (schema + native handlers + scheduler + per-user SSE), `TASKS_BACKEND=vikunja`. Purely additive — behavior identical to today.
2. **Backup:** `pg_dump` the `app` DB; snapshot the `vikunja` DB.
3. **Dry-run** migration; reconcile counts vs the Vikunja UI.
4. **Freeze writes** (brief window — single user): proxy returns 503 on task `POST/PUT/DELETE`, or a read-only banner.
5. **Run migration** `--include-done`. Run the verification gate (§ below).
6. **Flip:** set `TASKS_BACKEND=postgres`, `rollout restart`. This atomically switches the API to Postgres **and** enables the scheduler/per-user SSE. Client unchanged. Unfreeze.
7. **Soak 3–7 days** with Vikunja running but idle (warm rollback target). Delete nothing yet. The Vikunja webhook + `broadcast` stay in place but receive no traffic.
8. **Retire Vikunja** (Phase 5 below).

**Verification gate (after step 5 and again after step 6):** counts `tasks/projects/labels/task_reminders/task_labels` per project vs Vikunja; harness spot-checks of `/api/vikunja/{projects,tasks?sort_by=due_date…,projects/{id}/tasks,labels}` confirming zero-sentinel on empty dates, `reminders:[{reminder}]`, `labels:[{id,title}]`; **golden-diff** a few tasks' Vikunja JSON vs Postgres JSON; headless UI smoke at the real hostname — Inbox loads, Upcoming groups, and a **tasklist widget whose `projectId` came from a saved layout still resolves** (proves ID preservation).

**Rollback:**
- Steps 1-5 (pre-flip): trivial — nothing user-visible changed; unfreeze; re-running migration is safe.
- Step 6 (post-flip, pre-soak): set `TASKS_BACKEND=vikunja` + `rollout restart`. Vikunja holds the exact pre-freeze state; scheduler goes dormant; webhook/`broadcast` resume. Cost = post-flip Postgres edits (kept ~zero by a tight gate with the banner up until it passes). **Document that post-flip edits are lost on this path.**
- During soak: env-flip rollback still works but loses all post-flip edits → break-glass only. Prefer fix-forward (restore `app` from step-2 `pg_dump`, re-migrate).
- Post-retirement: no clean rollback — retirement only happens after a successful soak; step-2 backups are the last resort.

**Vikunja retirement (Phase 5, after soak):** `kubectl delete -f k8s/20-vikunja.yaml` (Deployment+Service+`vikunja-files` PVC) and remove the file; delete `vikunja-secret`; remove `VIKUNJA_*` + `WEBHOOK_SECRET` from `reminders-app-env`; delete `app/server/vikunja.js`; remove the webhook route (`index.js:29`), `handleWebhook` + `broadcast` from `events.js`; drop the `vikunja` DB after a final dump (`psql -c 'DROP DATABASE vikunja;'`) and remove `CREATE DATABASE vikunja` from the `pg-init` ConfigMap; delete the external Vikunja webhook registration. **Keep:** `pg` StatefulSet/Service/`reminders-pg` secret, app Deployment/Service, `40-httproute.yaml`, `50-networkpolicy.yaml` (the `tier: backend` selector still correctly fences `pg`), OIDC/CalDAV/layouts and their env. Update `README.md` and the two memory files (`reminders-app-project.md`: tasks are now Postgres-native keyed by OIDC `sub`, shared-account leak fixed, the `POST /tasks/{id}` read-modify-write gotcha is obsolete, reminders via in-app scheduler; `reminders-app-test-harness.md`: the `vk()` exec snippet still works on the same paths, now Postgres-served).

## 9. Riskiest parts and how to de-risk

- **R1 — Native API field-fidelity / partial-update semantics (highest, correctness).** Any field the client reads (`done`, `due_date` sentinel, `reminders:[{reminder}]`, `labels:[{id,title}]`, `repeat_*`) that's wrong or zeroed silently breaks UX. *De-risk:* the single `serializeTask` source of truth; **golden-response diff** (same task via Vikunja vs Postgres) during dual-run; unit tests; the `TASKS_BACKEND` flag keeps the proxy one env-flip away.
- **R2 — Migration data loss / ID renumbering (highest, ops).** Renumbering projects breaks every saved dashboard (`layout_json.projectId`). *De-risk:* preserve ids (`GENERATED BY DEFAULT` + `setval`); `--dry-run` count reconciliation; `pg_dump` backup; write-freeze guarantees idempotent convergence (never run against a live-writing backend); gate includes a saved-layout-referenced project resolving in the UI.
- **R3 — Recurrence `repeat_mode` mapping (correctness).** The brief's description is swapped vs real Vikunja. *De-risk:* implement the real enum (1=monthly, 2=from-current); migration emits a `repeat_mode<>0` sample report; unit matrix incl. Go month-overflow; end-to-end spot-check a recurring task before flip.
- **R4 — Reminder boot-flood after migration.** The scheduler's first tick would fire every past reminder. *De-risk:* migration stamps `fired_at=now()` on `remind_at<=now()`; only future reminders stay pending.
- **R5 — Multi-tenancy must actually close.** *De-risk:* composite FKs + `WHERE user_id` on every query + 404-not-403 + per-user SSE registry; test with two distinct subs that neither can read/attach the other's rows.
- **R6 — At-most-once SSE misses a buzz while disconnected.** *De-risk:* durable overdue chip in the task list is the source of truth; optional `/api/reminders/recent` backfill on reconnect.
- **R7 — Post-flip edits lost on env-flip rollback.** *De-risk:* keep the verification gate to minutes with the maintenance banner up until it passes; treat soak rollback as break-glass and prefer fix-forward from the step-2 dump.

**Cross-cutting de-risking:** dual-run via `TASKS_BACKEND` (byte-compatible wire contract), migration `--dry-run` before the real run, `pg_dump` + Vikunja DB snapshot before the flip, a soak window with Vikunja kept warm, and a tight verification gate run both before and after the flip.
